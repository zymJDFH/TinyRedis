# TinyRedis 工业化实现方案（v1.0）

更新时间：2026-04-13

## 1. 文档目标

本文档用于将 TinyRedis 从“学习型可运行原型”推进到“可上线、可维护、可扩展”的工业级实现。  
核心目标：

- 明确工业级版本的需求边界与质量标准。
- 用高内聚低耦合的架构方式组织代码。
- 提供可执行的时间规划、阶段验收与风险控制。
- 形成具有工程含金量的成果物组合（代码、测试、压测、故障演练、设计文档）。

## 2. 工业级目标定义

## 2.1 目标边界

- 目标一：工业可用子集（Production-ready Core）。
- 目标二：持续逼近 Redis 主线能力，而不是一次性复制全部 300+ 命令。
- 非目标：短期内完全对齐 Redis Cluster、Modules、Lua、ACL 全能力。

## 2.2 质量目标（SLO）

| 维度 | 目标 |
| --- | --- |
| 正确性 | 核心命令语义与 Redis 保持一致，关键行为有回归测试 |
| 可用性 | 单实例长稳运行 72h 无崩溃、无内存泄漏 |
| 延迟 | 本机回环下 `GET/SET` p99 延迟可控（基线持续下降） |
| 吞吐 | 保持并超越当前 `epoll LT` 基线，并建立可复现实验 |
| 持久化 | AOF `everysec` 可恢复，异常退出后数据损失窗口 <= 1s |
| 可维护性 | 关键模块职责单一，跨模块只走显式接口 |
| 可观测性 | 具备基础指标、结构化日志、慢命令采样 |

## 3. 当前基线与差距（截至 2026-04-13）

## 3.1 当前已完成

- SDS 基础动态字符串实现。
- DICT 基础 `set/get/erase` API。
- RESP2 基础解析与编码。
- 命令链路：`PING/SET/GET/DEL/EXISTS/INCR`。
- 单线程 `epoll`（LT）服务循环，可被 `redis-cli` 连接。
- 基础单元测试框架与四组测试用例。

## 3.2 对比 Redis 源码的完成度估算

说明：以下百分比用于路线规划，不代表绝对精确值。

| 模块 | 当前状态 | 对标 Redis 现状 | 完成度 |
| --- | --- | --- | --- |
| 网络事件循环 | 单线程 `epoll LT` 可用 | Redis 具备成熟事件框架与复杂连接状态机 | 30% |
| 协议层 | RESP2 关键类型可用 | Redis 支持更完整协议处理与防御性策略 | 40% |
| 命令系统 | 6 个命令 | Redis 7.2 命令体系规模巨大 | 8% |
| 数据结构 | SDS/DICT 基础版 | Redis 有对象系统与多编码策略 | 12% |
| 过期系统 | 未完成 | Redis 具备惰性 + 主动过期 | 0% |
| 持久化 | 未完成 | Redis AOF/RDB 完整链路 | 0% |
| 复制高可用 | 未完成 | Redis 主从复制与故障转移成熟 | 0% |
| 工程化质量 | 有单测，无 E2E/模糊/故障注入 | Redis 工程化体系完善 | 15% |

综合估算：

- 对照 TinyRedis 当前 roadmap：约 50%-55%。
- 对照“工业级 Redis 子集”目标：约 20%-25%。
- 对照 Redis 全量能力：约 8%-12%。

## 4. 需求分析

## 4.1 功能需求（FR）

- FR-1：支持 String/Hash/List/Set/ZSet 五大类型的核心命令子集。
- FR-2：支持 Key 级 TTL（`EXPIRE/TTL/PTTL/PERSIST`）并保证语义正确。
- FR-3：支持 AOF 持久化与重启恢复。
- FR-4：支持基础主从复制（全量同步 + 增量传播）。
- FR-5：支持 `INFO`、`CONFIG GET`、`PING` 等基础运维命令。

## 4.2 非功能需求（NFR）

- NFR-1：接口稳定，模块边界清晰，避免隐式跨层访问。
- NFR-2：命令执行链路具备可追踪性与异常可定位性。
- NFR-3：核心路径有性能基准，改动后可做回归对比。
- NFR-4：关键模块有单测、集成测试、端到端测试。
- NFR-5：可在 CI 中稳定构建与测试。

## 4.3 约束与假设

- 以单线程 Reactor 为主线，优先语义正确性与稳定性。
- 保持与当前代码风格兼容，渐进重构，不做一次性推倒重写。
- 先做“工业可用子集”，再拓展到更多 Redis 特性。

## 5. 可行性分析

## 5.1 技术可行性

| 方向 | 当前基础 | 难点 | 可行性结论 |
| --- | --- | --- | --- |
| 命令扩展 | 解析与分发主链路已通 | 命令语义边界多 | 高，可迭代推进 |
| TTL 系统 | 已有读写访问入口 | 时钟一致性、删除时机 | 高，先惰性后主动 |
| 对象系统 | SDS/DICT 已有雏形 | 对象编码与生命周期 | 中高，需先抽象 |
| 持久化 AOF | 事件循环与命令执行已稳定 | 崩溃一致性、重放性能 | 中，需要严格测试 |
| 复制能力 | 网络层可复用 | 状态机与一致性复杂 | 中，建议后置 |

## 5.2 资源可行性

- 单人开发可行，但必须阶段化交付。
- 建议按 2 周一个 Sprint 推进，避免需求扩张导致失控。
- 每个 Sprint 固定输出：功能代码 + 测试 + ADR + 压测结果。

## 5.3 风险与应对

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 功能膨胀 | 周期失控 | 先锁定工业子集，命令按优先级分层 |
| 性能先行导致语义回归 | 隐蔽 bug 增多 | 先 correctness，再优化 |
| 模块边界不清 | 维护成本陡增 | 强制接口分层与依赖规则 |
| 缺少稳定性验证 | 上线风险高 | 增加长稳压测、故障注入、Sanitizer |

## 6. 架构原则与模块设计（高内聚低耦合）

## 6.1 设计原则

- 单一职责：每个模块仅负责一个明确领域。
- 接口隔离：模块间只通过头文件接口交互，不共享内部状态。
- 依赖单向：`net -> protocol -> command -> db/object -> core`。
- 可替换性：数据结构和存储策略通过抽象层可替换。
- 可测试性：模块内部逻辑可在无网络环境下独立测试。

## 6.2 推荐模块边界

- `core`：SDS、DICT、时间工具、内存工具。
- `object`：RedisObject、编码策略、类型系统。
- `db`：键空间、过期字典、键访问 API。
- `command`：命令元数据、参数校验、执行器。
- `protocol`：RESP 编解码与协议错误处理。
- `net`：连接管理、事件循环、读写缓冲。
- `persistence`：AOF 写入、刷盘策略、重放恢复。
- `replication`：主从同步状态机、复制流传输。
- `observability`：metrics、日志、慢命令与诊断。

## 6.3 依赖约束

- `command` 不直接操作 socket。
- `net` 不包含业务语义判断。
- `db` 不关心协议格式，只接收结构化请求。
- `persistence` 通过命令传播接口接入，不侵入命令实现细节。

## 7. 时间规划（建议 24 周）

## 阶段 P0（第 1-2 周）：工程化基线

- 目标：重整目录、接口边界、统一错误码与日志模型。
- 产出：模块依赖图、编码规范、CI 基线、ASan/UBSan 跑通。
- 验收：`ctest` + Sanitizer 绿灯。

## 阶段 P1（第 3-6 周）：对象系统 + TTL

- 目标：引入 `RedisObject`，DB 改为对象存储，完成惰性过期。
- 产出：`EXPIRE/TTL/PTTL/PERSIST`，过期语义测试。
- 验收：核心命令 + TTL 端到端测试通过。

## 阶段 P2（第 7-12 周）：数据结构与命令族

- 目标：完成 List/Hash/Set/ZSet 核心命令子集。
- 产出：命令表驱动分发、类型检查、错误语义对齐。
- 验收：命令回归集稳定，语义与 Redis 对照测试通过。

## 阶段 P3（第 13-17 周）：持久化（AOF）

- 目标：支持 `appendonly yes`、`always/everysec/no` 策略。
- 产出：AOF 重放恢复、崩溃恢复测试、数据一致性报告。
- 验收：异常退出后可恢复，数据损失窗口符合目标。

## 阶段 P4（第 18-21 周）：复制基础能力

- 目标：主从复制最小闭环（全量 + 增量）。
- 产出：复制握手、传播机制、断线重连恢复。
- 验收：主从一致性测试通过。

## 阶段 P5（第 22-24 周）：稳定性与性能冲刺

- 目标：长稳、压测、慢命令、可观测性补齐。
- 产出：基准报告、火焰图、性能优化 ADR。
- 验收：72h 稳定运行，核心性能指标达标。

## 8. 阶段验收与发布门禁

- 功能门禁：核心需求全部有测试覆盖。
- 质量门禁：无 blocker 级缺陷，Sanitizer 无新告警。
- 性能门禁：回归后吞吐与延迟不劣化。
- 运维门禁：日志、指标、诊断命令可用。
- 文档门禁：每个非平凡决策必须补充 ADR。

## 9. 含金量成果物清单

- 系统架构图与模块依赖图。
- 关键 ADR（对象模型、TTL、AOF、复制）。
- 可重复压测脚本与基准结果报告。
- 故障演练记录（崩溃恢复、断连恢复、慢命令定位）。
- 一套可公开展示的工程化测试矩阵。

## 10. 本周可执行计划（立刻开工）

- 任务 1：定义 `RedisObject` 与 `DB` 抽象接口。
- 任务 2：落地惰性过期与 `EXPIRE/TTL/PTTL/PERSIST`。
- 任务 3：补齐命令层 + 网络层 E2E 测试（半包、粘包、pipeline）。
- 任务 4：补一篇 ADR：对象生命周期与内存所有权模型。

完成标准：

- 新增功能全量可测。
- `ctest --output-on-failure` 全绿。
- 形成可审阅设计文档与基准记录。
