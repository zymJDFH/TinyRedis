# TinyRedis Roadmap

本文档用于记录 TinyRedis 的阶段目标、当前状态和下一步任务拆分。项目目标不是一次性复刻完整 Redis，而是沿着“可运行 -> 可测试 -> 可演进 -> 可展示”的路径，逐步完成一个具备真实工程味道的 C++ Redis 内核子集。

## 1. 项目定位

TinyRedis 当前更适合被定义为：

- 一个基于 C++17 的 Redis 内核学习与工程化实践项目
- 一个围绕单线程事件循环、协议解析、命令执行、持久化和复制展开的系统项目
- 一个强调模块清晰度、测试基线和可持续演进的个人核心项目

当前阶段不追求“命令数量尽可能多”，而更关注：

- 核心链路行为正确
- 模块边界稳定
- 文档、测试和性能基线完整
- 后续每一阶段都能独立讲清楚设计取舍

## 2. 当前状态

已完成：

- 单线程 `epoll` 事件循环与 TCP 网络服务
- RESP2 编解码与连续消息解析
- 命令解析、参数校验与分发链路
- String / Hash 基础数据模型与命令支持
- TTL 元信息、惰性过期和主动过期
- AOF 追加、回放、同步重写和后台重写
- 简化版 master/replica 全量同步与命令传播
- `INFO` 指标输出
- 单元测试、AOF 测试、E2E 测试与性能基线

当前短板：

- 数据结构目前覆盖 String 和基础 Hash，仍缺少 List / Set / ZSet 等更完整的复合类型
- 文档与目录现状存在少量不一致，项目完成度观感还有提升空间
- 复制能力尚未进入 backlog / 部分重同步阶段
- 容错、回归覆盖和 benchmark 自动化还可以继续补强

## 3. 阶段路线

### Phase 1: 工程化收口 v1.0

目标：

- 让仓库结构、文档入口、能力边界、测试方式全部对齐
- 形成一个可以稳定展示和持续迭代的基础版本

任务：

- 补齐路线文档、能力矩阵和文档索引
- 清理 README 中的目录说明与实际文件差异
- 统一“当前支持 / 暂未支持 / 后续计划”的表述
- 补一版命令支持矩阵
- 补一版测试矩阵和性能测试说明

完成标准：

- 新同学首次打开仓库，5 分钟内可以理解项目目标、现状和下一步
- README、设计文档、路线文档之间没有明显冲突
- `ctest` 和 benchmark 入口清晰可复用

### Phase 2: 选择一条核心特性主线

目标：

- 聚焦一个最能体现项目深度的方向，避免功能碎片化推进

推荐只选一条主线：

- 数据结构扩展：优先 `List`
- 持久化增强：补 `RDB snapshot`
- 复制增强：补 backlog 与部分重同步骨架

建议优先级：

1. 如果目标偏“工程稳步迭代”，优先做 `List`
2. 如果目标偏“面试和系统深度”，优先做 `RDB` 或复制增强

完成标准：

- 对外能明确描述这个阶段解决了什么问题
- 设计文档补充该特性的模块边界、数据流和测试方案
- 至少新增一组 E2E 或回归测试覆盖核心行为

### Phase 3: 稳定性与可观测性增强

目标：

- 让 TinyRedis 更像一个真实服务，而不只是功能演示程序

任务候选：

- 更细的错误码和异常路径测试
- 启动恢复、AOF 故障和 rewrite 边界测试
- 复制状态指标补充
- benchmark 结果归档与对比流程标准化

完成标准：

- 关键失败路径有测试或可重复演示用例
- 能输出一份“优化前后”性能对比

### Phase 4: 展示与沉淀

目标：

- 把项目从“自己知道做了什么”升级为“别人也能快速看懂价值”

任务：

- README 首页补项目亮点、能力矩阵、性能摘要
- 输出 1 到 2 篇围绕架构或演进过程的文章
- 准备 3 个固定演示场景：基础 KV、AOF 恢复、主从复制或重写 AOF

完成标准：

- 项目可直接用于简历、面试或博客展示
- 每个核心模块都能用一句话讲清职责和设计取舍

## 4. 接下来两周的建议节奏

### Week 1

- 完成文档收口
- 补命令支持矩阵
- 补测试矩阵
- 整理一个公开的 issue 列表或任务清单

### Week 2

- 启动唯一一条主线特性开发
- 先写设计草图，再拆实现任务
- 每完成一个子能力就补对应测试

## 5. 近期任务清单

建议按下面顺序推进：

1. 文档对齐：README、design、roadmap
2. 命令矩阵：已支持、部分支持、未支持
3. 测试矩阵：模块对应测试和覆盖范围
4. 选主线：`List`、`RDB`、复制增强三选一
5. 设计评审：为主线特性补一份简要设计说明
6. 实现与验证：代码、测试、benchmark 一起推进

## 6. 暂不优先事项

以下内容短期内不建议同时展开：

- 一次性补多个复合数据结构
- 事务、Lua、集群等跨度很大的 Redis 高级能力
- 在主线尚未明确前分散做大量小命令

原因是这些工作会迅速扩大维护面，削弱当前项目已经建立起来的模块清晰度和完成度。
